# AI 应用能力展示

## 一、背景：从"不会 Rust 和 React"到"全栈独立开发"

DbPaw 是一个从零开始用 Rust 开发的数据库管理工具，开源一个月获得 100+ Star。

在项目开始之前，有两个事实：

1. **我不会 Rust**
2. **我不会 React**

但项目最终完成了，而且代码质量足够支撑开源。

怎么做到的？

**AI。**

但不是那种"AI帮我写代码"的简单理解，而是在这个过程中，我形成了完全不同的开发认知。

---

## 二、幻想破灭：AI 不是万能的

项目刚开始的时候，我有一个天真的想法：

> 既然 AI 什么都会，那我是不是可以让 AI 直接把整个项目写出来？

于是我开始疯狂尝试：

- 让 AI 生成 React UI
- 让 AI 写 Rust backend
- 让 AI 设计数据库连接模块
- 让 AI 设计整个项目结构

刚开始看起来效果还不错，AI 确实能很快生成一堆代码。

**但问题很快就出现了。**

### 第一个问题：没有架构

AI 每次生成的代码都是**局部正确**的。

一个页面能跑，一个组件能用，一个函数能执行。

但整体项目却越来越混乱：

- 文件结构不一致
- 状态管理混乱
- 组件之间耦合严重
- Rust 和前端的接口设计反复变化

换句话说：

> **AI 可以写代码，但不会替你做系统架构。**

### 第二个问题：调试成本极高

AI 写的代码运行报错，怎么办？

1. 手动修代码
2. 继续让 AI 改

如果继续让 AI 改，就会进入一个死循环：

```
报错 → 贴错误日志 → AI修改代码 → 出现新的错误 → 继续修改
```

很快就会发现：

- Token 消耗巨大
- 时间消耗巨大
- 项目结构越来越混乱

最后只得到一堆**半成品代码**。

### 第三个问题：代码可以跑，但不知道有没有 bug

这是最可怕的问题。

AI 生成的代码，很多时候看起来是对的，跑起来也是对的。

**但真的对吗？**

边界条件呢？异常分支呢？并发安全呢？

我发现一个很现实的问题：

> **没有测试，我根本不敢相信 AI 写的代码。**

---

## 三、第一个认知转变：测试是 AI 的"质量守门员"

经历了这些失败之后，我意识到一件事情：

**如果要让 AI 输出可用的代码，必须有测试来验证。**

于是我给自己的开发流程加了一条铁律：

> **AI 生成的每个功能模块，必须有对应的单元测试。**

### 具体做法

**第一步：测试先行**

在让 AI 生成功能代码之前，先让它生成测试用例。

比如实现一个 SQL 解析器：

```
先写测试：
- 正常 SQL 解析测试
- 边界条件测试（空值、超长字符串）
- 异常 SQL 测试（语法错误）
- 性能测试（大 SQL 解析时间）
```

然后再写代码。

**第二步：测试驱动迭代**

AI 生成的代码，用测试去验证。

测试失败 → 反馈给 AI → 修正代码 → 再次测试

**第三步：覆盖率检查**

使用 `cargo tarpaulin` 检查 Rust 代码覆盖率，用 `vitest` 检查前端代码覆盖率。

### 结果

| 指标 | 变化 |
|-----|------|
| 测试覆盖率 | 30% → 85% |
| AI 代码 Bug 率 | ~30% → < 5% |
| 修复一个 Bug 的时间 | 平均 30 分钟 → 平均 10 分钟 |

更重要的是，我形成了一个信念：

> **测试不是可选项，是 AI 开发的必选项。**

没有测试的 AI 代码，就是一颗定时炸弹。

---

## 四、第二个认知转变：AI 抹平了前后端边界

在传统开发模式中，前后端是两个不同的技能栈：

- 后端：Java、Go、Rust...
- 前端：React、Vue、CSS...

我之前的定位是**后端开发**。

前端？不太会。

但在这个项目中，我发现：

> **AI 让技术栈的边界变得模糊了。**

### 前端能力的快速补齐

不会 React？没关系。

不会 CSS？没关系。

不会状态管理？没关系。

**AI 会。**

关键是，AI 不仅能写代码，还能解释代码。

在开发过程中，我做了几件事：

**1. 让 AI 生成组件代码，同时解释实现逻辑**

```
AI: "这个组件使用了 React Hooks 的 useState 来管理状态..."
我: "为什么不用 useReducer？"
AI: "useReducer 更适合复杂状态，这个场景 useState 足够..."
```

通过这种方式，我在使用 AI 的同时，也在学习。

**2. 让 AI 推荐最佳实践**

```
我: "这个表单应该用什么组件库？"
AI: "推荐 shadcn/ui，原因：
    1. 可定制性强
    2. TypeScript 支持好
    3. AI 生成代码时质量更高"
```

**3. 前后端类型同步**

最大的痛点：前后端类型定义不一致。

AI 的解决方案：

```
1. 后端定义 Rust struct
2. 使用 ts-rs 库自动生成 TypeScript 类型
3. 前端直接使用生成的类型
```

类型安全问题，从运行时检查变成了编译时检查。

### 结果

项目结束后，我可以自信地说：

> **我现在有全栈能力了。**

不是因为我学会了所有前端技术，而是因为：

1. 我理解了前端的核心概念
2. 我知道如何用 AI 高效完成前端开发
3. 我知道如何设计前后端协作的架构

## 五、开发流程的重构

经历了这些，我的开发流程完全变了。

### 传统开发

```
需求 → 设计 → 编码 → 测试
```

### AI 时代开发

```
需求
 ↓
架构设计（人）
 ↓
写 Plan 文档（AI 辅助）
 ↓
生成代码（AI）
 ↓
写单测（AI 辅助）
 ↓
代码 Review（人）
 ↓
运行测试（自动化）
 ↓
迭代优化
```

核心变化：

| 环节 | 传统模式 | AI 模式 |
|-----|---------|--------|
| 编码 | 人写 | AI 生成 |
| 测试 | 可选 | 必须有 |
| Review | 看时间 | 每个 PR 必须看 |
| 架构 | 边写边想 | 先设计再生成 |

**人从"写代码"变成了"设计系统 + 审核代码"。**

## 六、遇到的问题与应对

### 问题一：AI 幻觉

AI 有时候会"一本正经地胡说八道"。

比如：

- 引用不存在的 API
- 使用过时的语法
- 忽略边界条件

**应对：**

1. 所有 AI 生成的代码必须跑通测试
2. 不信任未经验证的代码
3. 使用 Clippy、ESLint 等工具辅助检查

### 问题二：Rust 借用检查器

Rust 的所有权系统是 AI 的噩梦。

AI 经常写出编译不通过的代码：

```
error[E0382]: borrow of moved value: `x`
```

**应对：**

1. 让 AI 先理解 Rust 所有权规则
2. 使用 `cargo check` 快速验证
3. 建立 Rust 代码模板库，减少重复错误

### 问题三：前端样式问题

AI 对 CSS 的理解有时候很奇怪。

生成的样式：

- 有时候不生效
- 有时候优先级混乱
- 有时候响应式布局有问题

**应对：**

1. 使用 TailwindCSS 减少手写 CSS
2. 使用 shadcn/ui 等成熟组件库
3. 人眼检查 UI 效果

## 八、日常 AI 能力提升

为了更好地使用 AI，我做了几件事：

**1. 积累有效的 Prompt 模板**

针对不同场景，整理了可复用的 Prompt。

**2. 探索 AI 工具链**

- Cursor：AI IDE
- Windsurf：另一个 AI IDE
- GitHub Copilot：代码补全

不同工具有不同的优势。

**3. 建立个人知识库**

记录：

- AI 生成代码的常见问题
- 有效的 AI 协作工作流
- 测试用例模板

## 九、总结

通过 DbPaw 项目，我形成了自己的 AI 开发方法论：

```
"AI 辅助 + 测试验证 + 人工把控"
```

**两个核心认知：**

1. **测试是 AI 的"质量守门员"**
   - 没有测试，AI 输出的代码不可信
   - 测试覆盖率是 AI 开发的核心指标

2. **AI 抹平了技术栈边界**
   - 从"后端开发"变成了"全栈开发"
   - 核心能力是"设计架构 + 使用 AI + 验证结果"

**最终结果：**

一个不会 Rust 和 React 的人，用 AI 完成了一个质量足够开源的项目。

这不是 AI 的胜利，是"人 + AI"的胜利。

---

> **AI 可以替代写代码，但不能替代思考。**
